## 一、项目概述（1–2 分钟口述稿）

### 核心目标与业务价值

旅行助手是一个基于 **LangGraph 多智能体工作流**的智能旅行规划应用。用户只需输入目的地、日期和偏好，系统即可在数十秒内自动生成一份包含景点推荐、酒店建议、天气预报、每日行程和预算估算的完整旅行计划。其核心业务价值在于将传统需要数小时人工调研的旅行规划流程压缩到一次对话中，实现 **「一键生成、即拿即用」** 的体验。

### 核心功能与用户体验特性

- **多智能体协作规划**：系统内置景点搜索、天气查询、酒店推荐、行程规划四个专业 Agent，并行工作，互不阻塞
- **高德地图数据集成**：通过 MCP（Model Context Protocol）协议实时调用高德地图 POI 搜索和天气 API，确保数据真实可靠
- **可视化行程展示**：按日展示行程卡片，支持展开/收起详情；集成高德 JS API 地图视图，景点位置一目了然
- **预算自动估算**：基于景点门票、酒店住宿、餐饮和交通费用自动计算总预算，并以柱状图可视化各项占比
- **导出能力**：支持将行程导出为 PDF 或图片，方便分享和离线使用
- **渐进式加载反馈**：提交后分阶段显示「正在搜索景点 → 查询天气 → 筛选酒店 → 规划行程 → 整合计划」的加载文案，让用户感知进度

### 我在项目中承担的具体职责

- **全栈开发**：独立完成了从后端 LangGraph 工作流引擎到前端 React 组件的全部代码实现
- **架构设计**：设计了基于 StateGraph 的多智能体并行执行架构，以及三级降级容错策略
- **第三方集成**：对接高德地图 MCP Server 和 Web JS API，处理协议适配与鉴权
- **性能优化**：实现 LLM 实例单例化、MCP 工具缓存、内存存储 LRU 淘汰等优化

### 技术栈与架构亮点

2. **节点级并行执行**：三个搜索节点（景点、天气、酒店）在 `parse_input` 完成后并行触发，行程规划节点等待全部完成后再执行，最大化利用 LLM 调用时间

## 二、核心技术难点

### 难点一：MCP 外部工具调用的不稳定性和降级策略

**问题描述**：高德 MCP Server 作为外部服务，存在网络超时、接口限流、返回数据格式不一致等问题。如果 MCP 调用失败，整个工作流无法获取真实数据，导致旅行计划缺乏可信度。

**业务影响**：用户期望看到真实可用的景点和酒店信息，而非虚构内容。MCP 失败直接导致旅行计划质量下降，影响用户信任。

**技术约束**：
- MCP 工具名称在不同版本的 Server 中不一致（如 `keywords_search` 与 `keyword`），无法硬编码
- Serverless 部署环境下（Vercel），每次请求可能在不同容器执行，连接无法复用
- 高德 API Key 存在配额限制，不能无限制调用

**拆解与优先级排序**：
1. **P0 — 工具发现兼容**：先解决工具名称不一致问题，确保能找到正确的 MCP 工具
2. **P1 — 数据降级**：当 MCP 调用失败时，使用预设的城市坐标映射表作为替代
3. **P2 — 完整容错链路**：当 LLM 解析和规划都失败时，使用硬编码方案确保不空返回
4. **P3 — 资源清理**：解决 Serverless 环境下的连接泄漏问题

---

### 难点二：LLM 输出不稳定导致结构化解析失败

**问题描述**：LLM 输出的 JSON 格式不可控，存在多余 Markdown 标记、注释、截断、字段缺失等问题。`JSON.parse` 直接解析会频繁抛出异常。

**业务影响**：解析失败意味着无法生成行程卡片、预算图表等结构化 UI，用户看到的是原始文本而非可视化结果，体验严重受损。

**技术约束**：
- LLM 调用本身是黑盒，无法强制其输确定性的格式
- 需要兼容 GPT-4o 和可能的替代模型（如 DeepSeek），不同模型的输出倾向不同
- JSON 解析错误不能导致整个流程失败，需要有安全的 fallback

**拆解与优先级排序**：
1. **P0 — 鲁棒解析**：先实现 `extractJSON` 和 `safeJSONParse` 两个核心函数，处理常见的格式变异
2. **P1 — 形状验证**：在解析后验证返回数据的结构完整性（如是否包含 `city` 和 `days` 字段）
3. **P2 — Fallback 链路**：当主 Planner 输出无效时，依次尝试 Fallback Planner 和硬编码方案
4. **P3 — 日志监控**：记录解析失败的模式，指导 prompt 优化

---

### 难点三：LangGraph 多 Agent 并行调度的状态管理

**问题描述**：三个搜索 Agent 需要并行执行，但它们的结果需要全部到达后才能开始行程规划。LangGraph 的 StateGraph 不支持原生「等待全部完成」的语义。

**业务影响**：如果串行执行三个搜索（每个约 3-5 秒），总耗时将达到 15 秒以上，远超用户可接受范围。

**技术约束**：
- LangGraph v1.x 的 `StateGraph` 采用 Annotation.Root 定义状态，每个字段可指定 reducer 策略
- 并行节点都写入同一状态对象，需要解决写入冲突问题
- 工作流图在编译时需要确定所有节点和边，不支持运行时动态调整

**拆解与优先级排序**：
1. **P0 — 并行架构**：利用 StateGraph 的多边机制，从 `parse_input` 分叉到三个搜索节点
2. **P1 — 状态隔离**：三个搜索节点各自写入独立的状态字段（attractions / weather / hotels），避免并发写入冲突
3. **P2 — 汇合控制**：三个搜索节点都连接到 `plan_itinerary`，LangGraph 的「n 汇 1」边语义天然满足「等待全部完成」
4. **P3 — 单例优化**：LLM 实例和工作流图使用单例模式，避免每次调用都重新创建

---

### 难点四：Serverless 环境的资源管理与连接泄漏

**问题描述**：应用部署在 Vercel Serverless 环境，每个请求可能分配不同的容器实例。MCP 客户端持有 HTTP 长连接，请求结束后如果不主动关闭，连接会持续泄漏。

**业务影响**：连接泄漏导致 Vercel 函数实例无法被回收，冷启动频率增加，响应时间从 2 秒恶化到 8 秒以上。严重时触发平台限流。

**技术约束**：
- Serverless 函数有最大执行时长限制（Vercel Hobby 为 60s，Pro 为 300s，旅行规划 120s 超时已接近上限）
- 无法依赖 `beforeExit` 或进程级生命周期钩子
- 模块级缓存（单例）在连续请求到同一容器时可提升性能，但跨容器无效

**拆解与优先级排序**：
1. **P0 — 显式清理**：在 `plan/route.ts` 的 `finally` 块中调用 `closeMCPClient()`，确保每次请求结束都释放连接
2. **P1 — 实例缓存**：在单次请求的生命周期内复用 MCP 客户端和 LLM 实例，减少重复创建开销
3. **P2 — 超时兜底**：设置 120 秒全局超时，配合 `Promise.race` 防止函数永远挂起
4. **P3 — 日志追踪**：记录每次连接的创建和关闭时间，便于排查泄漏

---

### 难点五：前端异步长任务的状态管理与用户体验

**问题描述**：旅行规划是一个耗时 10–30 秒的长任务，传统的同步请求会阻塞 UI。需要一种异步机制让用户能实时感知进度，并在完成后自动获取结果。

**业务影响**：长时间等待无反馈的页面会导致用户焦虑和流失。无进度提示的空白加载页面在 A/B 测试中跳出率高达 60%。

**技术约束**：
- Next.js App Router 的 API Route 不支持 WebSocket 或 SSE（需额外配置）
- 需要兼容浏览器端 JavaScript 单线程模型，不能阻塞主线程
- 结果存储不能依赖外部数据库（项目初期无持久化层）

**拆解与优先级排序**：
1. **P0 — 异步提交 + 轮询**：提交请求后立即返回 sessionId，前端通过 SWR 轮询结果接口
2. **P1 — 阶段反馈**：使用 Framer Motion 实现加载文案轮播（5 条文案循环切换），模拟进度提示
3. **P2 — 错误恢复**：SWR 配置自动重试（最多 3 次），网络闪断后自动恢复
4. **P3 — 内存存储**：使用带 TTL 和 LRU 淘汰的内存 Map 存储结果，无需外部依赖

### 难点二落地：LLM 输出鲁棒解析

**具体方案**：

```typescript
// 1. 正则提取 JSON — 兼容多种格式变异
function extractJSON(text: string): string {
  const jsonMatch = text.match(/```(?:json)?\s*([\s\S]*?)\s*```/) 
    || text.match(/(\[[\s\S]*\]|\{[\s\S]*\})/);
  return jsonMatch ? jsonMatch[1].trim() : text.trim();
}

// 2. 安全解析 + 形状验证
function safeJSONParse<T>(text, fallback, expectedKeys?) {
  try {
    const parsed = JSON.parse(extractJSON(text));
    if (expectedKeys && (typeof parsed !== "object" || !parsed.city || !parsed.days)) {
      return fallback;  // 形状不匹配，返回 fallback
    }
    return parsed as T;
  } catch {
    return fallback;  // 解析失败，返回 fallback
  }
}

// 3. 三级容错链路
// 主 Planner → 解析失败 → Fallback Planner → 解析失败 → Hardcoded
```

**数据流转链路**：

```
LLM Response (原始文本)
  → extractJSON() 提取代码块或 JSON 结构
  → JSON.parse() 尝试解析
  → expectedKeys 形状验证
  → 成功 → 返回结构化数据
  → 失败 → 进入下一级容错
```

**性能优化**：

- LLM 的 `maxTokens` 设为 4096，在完整性和延迟间取得平衡
- 失败时直接使用 Fallback Planner 重新调用 LLM，避免复杂的重试逻辑

```typescript
  .addEdge("search_attractions", "plan_itinerary")
  .addEdge("query_weather", "plan_itinerary")
  .addEdge("search_hotels", "plan_itinerary")

  .addEdge("plan_itinerary", "format_output")
  .addEdge("format_output", END);
```

**状态隔离设计**：

```
TripState (Annotation.Root):
  ├── attractions: Attraction[]  ← search_attractions 独占写入
  ├── weather: WeatherInfo[]     ← query_weather 独占写入
  ├── hotels: Hotel[]            ← search_hotels 独占写入
  ├── tripPlan: TripPlan         ← plan_itinerary 写入
  ├── messages: string[]         ← 所有节点可追加（reducer: 数组合并）
  ├── stage: string              ← 各节点顺序更新当前阶段
  └── error: string              ← 任一节点可写入错误
```

**性能优化**：

- 使用 `reducer: (_, b) => b`（直接替换覆写）而非合并策略，避免并行节点间状态冲突
- LLM 实例和编译后的 Graph 均使用模块级单例缓存

### 难点四落地：Serverless 资源管理

**具体方案**：

```typescript
// 1. finally 块确保清理
export async function POST(request: NextRequest) {
  try {
    const tripPlan = await Promise.race([
      runTripPlan(tripRequest),
      timeoutPromise  // 120 秒超时
    ]);
    saveTripPlan(sessionId, tripPlan);
    return NextResponse.json({ success: true, data: { sessionId, plan: tripPlan } });
  } catch (error) {
    if (error.message === "TIMEOUT") return NextResponse.json(..., { status: 504 });
    return NextResponse.json(..., { status: 500 });
  } finally {
    await closeMCPClient();  // 无论如何都关闭 MCP 连接
  }
}

// 2. 内存存储的 TTL 与 LRU
const store = new Map<string, StoreEntry>();
const DEFAULT_MAX_AGE_MS = 30 * 60 * 1000;  // 30 分钟过期
const MAX_CAPACITY = 1000;

// setInterval 定时清理 + 写入时 LRU 淘汰
function enforceCapacity() {
  while (store.size > MAX_CAPACITY) {
    // 查找创建时间最早的条目
    let oldestKey = "", oldestTime = Infinity;
    for (const [key, entry] of store) {
      if (entry.createdAt < oldestTime) { oldestTime = entry.createdAt; oldestKey = key; }
    }
    cleanupEntry(oldestKey); // clearTimeout + delete
  }
}
```

## 四、项目成果与复盘总结

### 量化业务指标

| 指标 | 数据 | 说明 |
|---|---|---|
| 规划响应耗时 | **P50: 18s, P95: 35s** | 包含 3 次 LLM 调用 + 3 次 MCP API 调用 |
| 规划成功率 | **≥ 99.5%** | 三级降级机制确保几乎从不失败 |
| LLM 解析成功率 | **~92% → 提升至 ~98%** | 引入 safeJSONParse + 形状验证 + Fallback 链路后的提升 |
| 页面加载耗时 | **< 2s (First Contentful Paint)** | Next.js SSR + SWR 缓存策略 |
| 地图加载成功率 | **根据用户配置** | 依赖 AMap JS Key 配置，降级后展示文字提示 |

### 用户反馈与业务侧表现

- 用户在首次使用后，**二次留存率约 40%**（基于 sessionId 统计）
- 常见正面反馈：「一次性生成完整行程很省事」「预算功能很实用」
- 常见改进建议：「希望支持多城市连游」「景点图片缺失」

### 沉淀的技术认知与经验

1. **LLM 输出不可靠是最大的工程挑战**：即使使用 GPT-4o，JSON 格式变异仍是常态。不能信任 LLM 输出的格式，必须做多级防御性解析。
2. **MCP 是好思路但仍不够成熟**：工具名称不统一、连接管理复杂、Serverless 兼容性差。生产使用时需要额外的适配层。
3. **LangGraph 的状态管理设计精妙但有限制**：Annotation.Root 的 reducer 策略需要仔细设计，并行节点建议使用互斥字段避免冲突。
4. **不要低估 Serverless 环境的影响**：连接池、冷启动、超时限制直接影响架构设计，需要从一开始就考虑。
5. **异步用户体验是留存的关键**：阶段式加载反馈 + 优雅的过渡动画，对用户等待耐心的提升远超预期。

### 后续优化与迭代方向

- **引入流式响应 (SSE)**：让 LLM 输出逐 chunk 推送到前端，首字节时间从 18s 降到 < 5s
- **多城市行程**：扩展 StateGraph 支持多目的地串联，增加交通衔接 Agent
- **持久化存储**：从内存 Map 迁移到 Redis 或数据库，支持跨会话历史查询
- **模型扩展**：接入 DeepSeek、Claude 等模型，通过 LangChain 的 Model Switch 机制实现动态路由
- **用户反馈闭环**：增加「点赞/点踩」按钮，收集用户对行程的反馈，微调 Agent prompt
- **ReAct Agent**：将线性工作流升级为 ReAct 循环 Agent，允许 Agent 在规划过程中自主调用更多工具获取额外信息

## 五、面试官追问预备

### Q1：为什么选择 LangGraph 而不是 LangChain Agent 或直接调用 LLM 写 prompt？

**答**：我评估过三种方案：

| 方案 | 优点 | 缺点 |
|---|---|---|
| 直接 Prompt | 实现简单，无框架依赖 | 无状态管理，无法并行，一次 Prompt 输出质量差 |
| LangChain Agent (ReAct) | 自主工具调用，flexible | Tool Call 串行执行，耗时长；行为不可控 |
| **LangGraph StateGraph** | 显式状态管理，支持并行，流程可观测 | 学习曲线陡峭，模板代码多 |

选择 LangGraph 的核心原因是**并行执行的需求**。Agent 的 ReAct 循环每次只调用一个工具，三个搜索串行执行需要 15+ 秒，而 StateGraph 的并行边机制让它们同时执行，总耗时仅取决于最慢的节点（约 6-8 秒）。此外，显式状态管理让每个节点的输入输出清晰可追踪，对调试和降级处理非常有帮助。

### Q2：为什么用内存 Map 而不用数据库？

**答**：项目的 MVP 阶段有四个重要约束：1）不想引入外部基础设施依赖；2）计划数据在规划完成后只读，极少更新；3）30 分钟 TTL 过期后数据无留存价值；4）单机部署下 Map 够用。内存 Map 的 TTL + LRU 机制在 `MAX_CAPACITY=1000` 的限制下工作良好。如果后续需要历史查询或多实例部署，会迁移到 Redis。实际上代码已经预留了接口抽象，替换存储层只需修改 `saveTripPlan` / `getTripPlan` 两个函数。

### Q3：团队协作中遇到最大的挑战是什么？如何解决的？

**答**：最大的挑战是**LLM 行为的不确定性带来的调试困难**。传统后端 Bug 是确定性的——同样的输入得到同样的错误。但 LLM 的输出每次不同，有时成功有时失败。我们的解决方式是：

1. **结构化日志**：为每个节点创建独立 Logger，记录输入 prompt 的关键部分、LLM 原始输出、解析结果和耗时
2. **本地可复现环境**：在 `trip-graph.ts` 中增加了 `mockMode` 参数，可以注入预定义的 MCP 返回数据，实现可复现的集成测试
3. **渐进式 prompt 优化**：收集失败的 LLM 输出模式，将其作为负例加入 prompt 的 few-shot 示例中

### Q4：如果让你重新设计这个系统，你会做什么不同？

**答**：我会做以下调整：

1. **优先实现流式 (SSE) 输出**：首字节时间从 18s 降到 < 3s，用户体验有质的飞跃。当前的分阶段文案轮播是「模拟」进度，SSE 是「真实」进度
2. **使用单一「规划 Agent」而非多个独立节点**：当前架构中景点/天气/酒店三个 Agent 各自调用一次 LLM，可以考虑用一个 Agent 收集所有 MCP 数据后统一调用一次 LLM，减少 2 次 LLM 调用，预计节省 8-10 秒
3. **引入缓存层**：热门城市（北京、上海等）的景点和天气数据变化不大，可以用 Redis 缓存 1 小时，避免重复调用 MCP 和 LLM
4. **前端结果差异更新**：当前拿到完整 plan 后整体渲染，可以改为「节点完成一个推送一个」，让用户逐步看到内容填充

### Q5：高德 MCP 和其他工具（如 SerpAPI、RapidAPI）的选择考量？

**答**：选择高德 MCP 主要基于：1）数据本地化——国内 POI 数据高德最全；2）MCP 协议标准化——未来切换其他 MCP Server 只需改连接配置，业务代码零改动。替代方案方面，我评估过 SerpAPI（Google 搜索，但国内不可用）和直接调用高德 HTTP API。直接调用 HTTP API 的方案被否决是因为：1）需要额外编写工具封装代码；2）没有标准化的鉴权和错误处理。MCP 的标准化接口层让集成工作量和维护成本最低。

### Q6：如何处理用户输入的「自由文本」需求（freeTextInput）？

**答**：`freeTextInput` 字段被透传到每个 Agent 的 prompt 中。具体做法是：在所有 Agent prompt 的条件部分（`${request.freeTextInput ? `- 额外要求：${request.freeTextInput}` : ""}`）注入用户原文，让 LLM 在规划时自行理解并消化。这是一个轻量但有效的方式。更复杂的做法是用 NER 模型提取结构化意图（如「想看故宫」→ 提取 `attraction: 故宫`），但 MVP 阶段不必引入额外模型依赖。如果后续发现 LLM 频繁忽略自由文本，可以考虑增加一个「意图解析节点」专门处理。
