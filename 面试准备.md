## 一、项目概述与核心价值

### 1.1 项目定位与业务目标

旅行助手是一个基于 **LangGraph 多智能体协作** 的智能旅行规划应用。用户输入目的地、出行日期和个性化偏好，系统在 **10–30 秒**内自动生成包含景点推荐、酒店建议、天气预报、每日行程安排和预算估算的完整旅行计划。

**核心业务价值**：
- 将传统需要 **数小时** 人工调研的旅行规划流程压缩到一次对话中
- 实现 **「一键生成、即拿即用」** 的极致用户体验
- 通过多智能体并行协作，确保数据的真实性和规划的合理性

### 1.2 核心功能特性

| 功能模块 | 功能描述 | 技术实现 |
|---------|---------|---------|
| 多智能体规划 | 景点搜索、天气查询、酒店推荐、行程规划四个 Agent 并行工作 | LangGraph StateGraph + 并行边 |
| 实时数据集成 | 通过 MCP 协议调用高德地图 POI 搜索和天气 API | MCP Client + 工具发现机制 |
| 可视化行程展示 | 按日展示行程卡片，支持展开/收起详情 | React + Framer Motion |
| 地图视图 | 集成高德 JS API 地图，景点位置一目了然 | AMap JS API v2.0 |

| 导出能力 | 支持 PDF 和图片导出 | html2canvas + jsPDF |
| 渐进式加载 | 分阶段显示加载文案，让用户感知进度 | SWR 轮询 + 状态管理 |

### 2.2 状态流转设计

```
TripState (Annotation.Root)
├── input: TripRequest          ← 用户输入，parse_input 验证后写入
├── attractions: Attraction[]   ← search_attractions 独占写入
├── weather: WeatherInfo[]      ← query_weather 独占写入
├── hotels: Hotel[]             ← search_hotels 独占写入
├── tripPlan: TripPlan          ← plan_itinerary 写入
├── messages: string[]          ← 所有节点可追加 (reducer: 数组合并)
├── stage: string               ← 各节点顺序更新当前阶段
└── error: string               ← 任一节点可写入错误
```

**Reducer 策略设计**：
- `attractions` / `weather` / `hotels`：使用 `reducer: (_, b) => b`，直接覆写，避免并行写入冲突
- `messages`：使用 `reducer: (a, b) => [...a, ...b]`，数组合并
- `stage`：使用 `reducer: (_, b) => b`，顺序更新
- `error`：使用 `reducer: (a, b) => a || b`，保留首个错误

### 2.3 工作流节点设计

| 节点名称 | 职责 | 输入依赖 | 输出 |
|---------|------|---------|------|
| `parse_input` | 验证用户输入，提取城市、日期、人数 | 用户请求 | 标准化的 TripRequest |
| `search_attractions` | 调用高德 POI 搜索，获取景点列表 | city, freeTextInput | attractions[] |
| `query_weather` | 调用高德天气 API，获取未来天气 | city, travelDays | weather[] |
| `search_hotels` | 调用高德酒店搜索，获取酒店推荐 | city, budget | hotels[] |
| `plan_itinerary` | LLM 基于数据生成结构化行程 | attractions, weather, hotels | tripPlan |
| `format_output` | 格式化输出，计算预算占比 | tripPlan | 最终行程数据 |

## 三、关键技术实现

### 3.1 难点一：MCP 外部工具调用的不稳定性与降级策略

**问题背景**：高德 MCP Server 作为外部服务，存在网络超时、接口限流、工具名称不一致等问题。

**技术约束**：
- 工具名称在不同版本 Server 中不一致（`keywords_search` vs `keyword`）
- Serverless 环境下连接无法复用
- API Key 存在配额限制

**解决方案**：

```typescript
function findTool(tools: MCPClientTool[], namePattern: string): MCPClientTool | undefined {
  return tools.find(
    t => t.name === namePattern ||
         t.name.includes(namePattern) ||
         t.name.toLowerCase().includes(namePattern.toLowerCase())
  );
}

const CITY_COORDS: Record<string, { longitude: number; latitude: number }> = {
  北京: { longitude: 116.4074, latitude: 39.9042 },
  上海: { longitude: 121.4737, latitude: 31.2304 },
  广州: { longitude: 113.2644, latitude: 23.1291 },
};

function getCityCoord(city: string): { longitude: number; latitude: number } {
  if (CITY_COORDS[city]) return CITY_COORDS[city];
  const normalized = city.replace(/市|省|区/g, '');
  if (CITY_COORDS[normalized]) return CITY_COORDS[normalized];
  return CITY_COORDS['北京'];
}
```

**三级降级链路**：
1. **Level 1**：LLM Planner（基于真实 MCP 数据）
2. **Level 2**：Fallback Planner（简化 prompt，不依赖 MCP 数据）
3. **Level 3**：`generateHardcodedPlan()`（完全硬编码，基于 CITY_COORDS）

### 3.2 难点二：LLM 输出不稳定导致结构化解析失败

**问题背景**：LLM 输出的 JSON 格式不可控，存在 Markdown 标记、注释、截断等问题。

**解决方案**：

```typescript
function extractJSON(text: string): string {
  const jsonMatch = text.match(/```(?:json)?\s*([\s\S]*?)\s*```/)
    || text.match(/(\[[\s\S]*\]|\{[\s\S]*\})/);
  return jsonMatch ? jsonMatch[1].trim() : text.trim();
}

function safeJSONParse<T>(text: string, fallback: T, expectedKeys?: string[]): T {
  try {
    const parsed = JSON.parse(extractJSON(text));
    if (expectedKeys && typeof parsed === 'object' && parsed !== null) {
      const missingKeys = expectedKeys.filter(k => !(k in parsed));
      if (missingKeys.length > 0) return fallback;
    }
    return parsed as T;
  } catch {
    return fallback;
  }
}
```

**解析链路**：
```
LLM Response → extractJSON() → JSON.parse() → 形状验证 → 成功/失败
                              ↓失败              ↓失败
                         Fallback Planner    Hardcoded Plan
```

### 3.3 难点三：LangGraph 多 Agent 并行调度

**问题背景**：三个搜索 Agent 需要并行执行，但结果需要全部到达后才能开始行程规划。

**解决方案**：

```typescript
const workflow = new StateGraph(TripState)
  .addNode("parse_input", parseInput)
  .addNode("search_attractions", searchAttractions)
  .addNode("query_weather", queryWeather)
  .addNode("search_hotels", searchHotels)
  .addNode("plan_itinerary", planItinerary)
  .addNode("format_output", formatOutput)
  .addEdge(START, "parse_input")
  .addEdge("parse_input", "search_attractions")
  .addEdge("parse_input", "query_weather")
  .addEdge("parse_input", "search_hotels")
  .addEdge("search_attractions", "plan_itinerary")
  .addEdge("query_weather", "plan_itinerary")
  .addEdge("search_hotels", "plan_itinerary")
  .addEdge("plan_itinerary", "format_output")
  .addEdge("format_output", END);
```

**并行执行原理**：LangGraph 的「n 汇 1」边语义天然满足「等待全部完成」，三个搜索节点都连接到 `plan_itinerary`，只有当所有上游节点完成后才会执行。

### 3.4 难点四：Serverless 环境的资源管理与连接泄漏

**问题背景**：Vercel Serverless 环境下，每个请求可能分配不同容器，MCP 客户端持有 HTTP 长连接，不主动关闭会导致泄漏。

**解决方案**：

```typescript
export async function POST(request: NextRequest) {
  try {
    const tripPlan = await Promise.race([
      runTripPlan(tripRequest),
      timeoutPromise(120000),
    ]);
    saveTripPlan(sessionId, tripPlan);
    return NextResponse.json({ success: true, data: { sessionId, plan: tripPlan } });
  } catch (error) {
    if (error.message === "TIMEOUT") return NextResponse.json({ error: "请求超时" }, { status: 504 });
    return NextResponse.json({ error: "服务器错误" }, { status: 500 });
  } finally {
    await closeMCPClient();
  }
}

const store = new Map<string, StoreEntry>();
const DEFAULT_MAX_AGE_MS = 30 * 60 * 1000;
const MAX_CAPACITY = 1000;

function enforceCapacity() {
  while (store.size > MAX_CAPACITY) {
    let oldestKey = "", oldestTime = Infinity;
    for (const [key, entry] of store) {
      if (entry.createdAt < oldestTime) {
        oldestTime = entry.createdAt;
        oldestKey = key;
      }
    }
    cleanupEntry(oldestKey);
  }
}
```

**问题背景**：旅行规划耗时 10–30 秒，传统同步请求会阻塞 UI。

**解决方案**：

```typescript
export function useTripPlan(sessionId: string | null) {
  const { data, error, isLoading } = useSWR(

    fetcher,

  return { tripPlan: data?.success ? data.data : null, isLoading, isError: !!error };
}
```

## 四、性能优化策略

### 4.1 后端优化

| 优化项 | 实现方式 | 效果 |
|-------|---------|------|
| LLM 实例单例化 | 模块级缓存 ChatOpenAI 实例 | 减少重复创建开销约 500ms |

| 优化项 | 实现方式 | 效果 |
|-------|---------|------|
| SWR 轮询 | 智能重试 + 指数退避 | 减少无效请求，提升稳定性 |
| 分阶段加载 | Framer Motion 动画轮播 | 提升用户等待体验 |

- 空数据处理：MCP 返回空数组时自动降级

### 5.3 前端容错

- SWR 自动重试：最多 3 次，指数退避
- 错误边界：Next.js Error Component 捕获渲染错误
- 地图加载失败：降级为文字提示

## 七、项目复盘与总结

### 7.1 量化业务指标

| 指标 | 数据 | 说明 |
|-----|------|------|
| 规划响应耗时 | P50: 18s, P95: 35s | 3 次 LLM + 3 次 MCP 调用 |

| 页面加载耗时 | < 2s (FCP) | Next.js SSR + SWR |

### 7.2 技术认知与经验

1. **LLM 输出不可靠是最大工程挑战**：必须做多级防御性解析
2. **MCP 协议不够成熟**：工具名称不统一、连接管理复杂
3. **LangGraph 状态管理设计精妙**：reducer 策略需要仔细设计
4. **Serverless 环境影响架构设计**：连接池、冷启动、超时限制需从一开始考虑
5. **异步用户体验是留存关键**：阶段式加载反馈对用户耐心提升显著
6. **模块化拆分是维护基础**：单体文件拆分为独立模块后，迭代效率提升

### 7.3 后续优化方向

- **流式响应 (SSE)**：首字节时间从 18s 降到 < 5s
- **多城市行程**：扩展 StateGraph 支持多目的地串联
- **持久化存储**：从内存 Map 迁移到 Redis
- **模型扩展**：接入 DeepSeek、Claude 等模型
- **用户反馈闭环**：增加点赞/点踩按钮，微调 prompt
- **ReAct Agent**：升级为循环 Agent，自主调用工具

## 八、面试常问问题与解答

### Q1：为什么选择 LangGraph 而不是 LangChain Agent 或直接调用 LLM？

**答**：

| 方案 | 优点 | 缺点 |
|-----|------|------|
| 直接 Prompt | 实现简单，无框架依赖 | 无状态管理，无法并行，输出质量差 |
| LangChain Agent | 自主工具调用，灵活 | Tool Call 串行执行，耗时长，行为不可控 |
| **LangGraph** | **显式状态管理，支持并行，流程可观测** | 学习曲线陡峭，模板代码多 |

选择 LangGraph 的核心原因是**并行执行需求**。Agent 的 ReAct 循环每次只调用一个工具，三个搜索串行需要 15+ 秒；而 StateGraph 的并行边机制让它们同时执行，总耗时仅取决于最慢节点（约 6-8 秒）。此外，显式状态管理让调试和降级处理更清晰。

### Q2：为什么用内存 Map 而不用数据库？

**答**：MVP 阶段有四个约束：
1. 不想引入外部基础设施依赖
2. 计划数据在规划完成后只读，极少更新
3. 30 分钟 TTL 过期后数据无留存价值
4. 单机部署下 Map 够用

内存 Map 的 TTL + LRU 机制在 `MAX_CAPACITY=1000` 限制下工作良好。代码已预留接口抽象，替换存储层只需修改 `saveTripPlan` / `getTripPlan` 两个函数。

### Q3：如何处理 LLM 行为的不确定性带来的调试困难？

**答**：
1. **结构化日志**：为每个节点创建独立 Logger，记录 prompt 关键部分、LLM 原始输出、解析结果和耗时
2. **本地可复现环境**：节点函数支持 `mockMode` 参数，可注入预定义 MCP 返回数据
3. **渐进式 prompt 优化**：收集失败的 LLM 输出模式，作为负例加入 few-shot
4. **LangSmith 全链路追踪**：设置可配置的 run_name 与 metadata，在 UI 中检索调试

### Q4：如果让你重新设计这个系统，你会做什么不同？

**答**：
1. **优先实现流式 (SSE) 输出**：首字节时间从 18s 降到 < 3s，真实进度反馈
2. **使用单一规划 Agent**：当前三个搜索 Agent 各调用一次 LLM，可合并为一个 Agent，减少 2 次调用
3. **引入缓存层**：热门城市景点和天气数据缓存 1 小时，避免重复调用
4. **前端结果差异更新**：节点完成一个推送一个，让用户逐步看到内容填充

### Q5：高德 MCP 和其他工具的选择考量？

**答**：选择高德 MCP 主要基于：
1. **数据本地化**：国内 POI 数据高德最全
2. **协议标准化**：未来切换其他 MCP Server 只需改连接配置，业务代码零改动

评估过 SerpAPI（国内不可用）和直接调用高德 HTTP API（需额外封装，无标准化鉴权）。MCP 的标准化接口层让集成工作量和维护成本最低。

### Q6：如何处理用户的自由文本需求？

**答**：`freeTextInput` 字段被透传到每个 Agent 的 prompt 中：
```
${request.freeTextInput ? `- 额外要求：${request.freeTextInput}` : ""}
```

这是轻量但有效的方式。更复杂的做法是用 NER 模型提取结构化意图，但 MVP 阶段不必引入额外模型依赖。如果后续发现 LLM 频繁忽略自由文本，可以增加「意图解析节点」。

### Q7：并行节点如何避免状态冲突？

**答**：通过状态隔离设计避免冲突：
1. 三个搜索节点各自写入独立状态字段（`attractions` / `weather` / `hotels`）
2. 使用 `reducer: (_, b) => b` 直接覆写，而非合并策略
3. `messages` 使用数组合并 reducer，允许各节点追加日志

这种设计确保每个节点的写入操作互不干扰。

### Q8：如何保证 Serverless 环境下的连接不泄漏？

**答**：
1. **finally 块显式清理**：在 `/api/trip/plan` 的 finally 块中调用 `closeMCPClient()`
2. **单次请求生命周期复用**：MCP 客户端和 LLM 实例在单次请求内复用
3. **超时兜底**：120 秒全局超时 + Promise.race 防止函数挂起
4. **日志追踪**：记录连接创建和关闭时间，便于排查

### Q9：前端为什么用 SWR 轮询而不是 WebSocket 或 SSE？

**答**：
1. **简单性**：Next.js App Router 的 API Route 原生不支持 WebSocket/SSE，需要额外配置
2. **可靠性**：轮询机制在网络不稳定时更健壮
3. **实现成本**：SWR 的轮询配置非常简单，只需几行代码
4. **降级兼容**：轮询对浏览器兼容性更好

如果后续需要更低延迟，可以升级到 SSE。

### Q10：如何保证测试的可靠性和可重复性？

**答**：
1. **Mock 模式**：节点函数支持注入预定义数据，实现可复现的集成测试
2. **边界测试**：覆盖空数据、异常格式、超时等场景
3. **CI 基线测试**：硬编码方案作为 always-pass 的基线测试
4. **Vitest 配置优化**：Windows 环境下使用 threads pool + 增大 testTimeout
